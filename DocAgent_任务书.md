# DocAgent —— 带自我反思的文档问答 Agent

> 项目任务书 / 开发规格说明
> 用途：个人项目，用于简历与面试展示「RAG + Agent」复合能力
> 配合 Claude Code / Codex 分阶段实现

---

## 0. 一句话定位

DocAgent 不是普通的「检索→拼 prompt→生成」RAG 问答机器人，而是一个会**自我评估检索质量、不够就改写 query 再查、实在查不到就老实说「资料里没有」**的文档问答 Agent。它把 Agent 的决策循环套在 RAG 外面，用来对抗普通 RAG 最大的毛病：检索得烂也硬答，导致幻觉。

对应的学术名词：**Corrective RAG (CRAG) / Self-RAG**（2024 年起的方向）。面试时能说出这个名字并讲清它解决的痛点，是加分项。

---

## 1. 为什么做这个（写进 README 的动机段）

普通 RAG 的死穴：
1. 检索一次就生成，检索结果差也硬塞进 prompt，模型只能编 → 幻觉。
2. 用户问法和文档措辞不一致时，向量检索召回差，但系统不会自救。
3. 答案不标来源，用户无法判断可信度。

DocAgent 的解法：在检索和生成之间加一个**评估 + 决策环**，让系统在「答」之前先判断「我查到的东西够不够答」，不够就自己想办法（改写 query 重查），还是不够就承认不知道，而不是硬编。

---

## 2. 核心机制（项目的灵魂，也是面试弹药库）

整个流程是一个带条件分叉和循环的状态机：

```
用户提问
   │
   ▼
[1] 检索 Retrieve  ── 从向量库召回 top-k 候选片段
   │
   ▼
[2] 评估 Grade ──── 让 LLM 判断每个片段与问题的相关性：相关 / 不相关 / 模糊
   │
   ▼
[3] 决策 Decide
   ├─ 有足够相关片段 ──────────────► [4] 生成 Generate
   ├─ 片段不足/不相关 且 未超重试上限 ─► [5] 改写 Query → 回到 [1]
   └─ 重试超上限仍不足 ────────────► [6] 兜底：回答「资料中未找到相关信息」
   │
   ▼
[4] 生成 Generate ── 基于通过评估的片段生成答案，并附来源引用
   │
   ▼
[7]（加分）自检 Self-check ── 校验答案是否真的基于检索内容，否则标注「可能不准确」
```

**关键设计点（每一个都是面试可被追问的取舍）：**
- 评估环节（Grade）：用 LLM 还是用相似度阈值？→ 选 LLM 打分，因为相似度高 ≠ 能回答问题；代价是多一次 LLM 调用，要在 prompt 里约束它只输出结构化结果（如 JSON 的 relevant: true/false）。
- 重试上限：必须设上限（建议 2 次），否则查不到会无限循环、烧钱。
- query 改写策略：让 LLM 基于「原问题 + 上轮为什么不相关」改写，而不是随机换词。
- 兜底优于硬答：宁可说「没有」，也不编 —— 这是整个项目的价值主张。

## 3. 技术选型（MVP 优先，别上重的）

| 组件 | 选型 | 理由 |
|------|------|------|
| 编排框架 | **LangGraph** | 带分叉+循环的状态机天然适合，比纯 LangChain 链式清晰 |
| 向量库 | **Chroma**（本地持久化） | 零配置、免部署，MVP 够用；别一上来上 Milvus |
| Embedding | 通义 text-embedding / bge-small-zh 等 | 中文文档优先选中文 embedding |
| LLM | DeepSeek API 或 通义千问 API | 已熟悉、便宜、够用 |
| 文档解析 | langchain 的 loader（pdf/markdown/txt） | MVP 先支持 1-2 种格式即可 |
| 接口 | 第一版 CLI；后续可选 FastAPI + Vue3 前端 | 先跑通逻辑，UI 是加分项不是必需 |
| 语言 | Python 3.10+ | —— |

**语料建议**：用你自己最熟的素材当知识库，demo 时讲得清楚。优先候选：novel2script 项目的代码与文档，或你 NLP 方向的几篇 paper。

## 4. 分阶段开发计划（适合喂给 Claude Code / Codex 逐段实现）

### 阶段 0：项目骨架（0.5 天）
- [ ] 初始化仓库，`requirements.txt`，`.env.example`（放 API key 占位）
- [ ] 目录结构：
  ```
  docagent/
    ingest.py        # 文档切分 + 向量化入库
    graph.py         # LangGraph 状态机定义
    nodes.py         # 各节点函数：retrieve / grade / rewrite / generate
    config.py        # 模型名、top_k、重试上限等参数
    main.py          # CLI 入口
  data/              # 放知识库源文件
  README.md
  ```
- [ ] 跑通一句 "hello"：能调通 LLM API、能调通 embedding API

### 阶段 1：最朴素 RAG（跑通闭环，1-2 天）
- [ ] `ingest.py`：加载文档 → 切块（chunk）→ embedding → 存进 Chroma
- [ ] `nodes.py` 的 retrieve：给定问题，召回 top-k 片段
- [ ] `nodes.py` 的 generate：把片段拼进 prompt，生成答案
- [ ] `main.py`：命令行输入问题 → 输出答案
- ✅ 里程碑：能像普通 RAG 一样问答（这一版故意先不带评估，作为对照基线）

### 阶段 2：加上评估 + 重查环（核心，2-3 天）
- [ ] grade 节点：对每个召回片段，让 LLM 判断相关性，输出结构化结果
- [ ] decide 逻辑：相关片段够 → 去生成；不够且未超上限 → 去改写；超上限 → 兜底
- [ ] rewrite 节点：基于原问题改写 query，回到 retrieve
- [ ] 用 LangGraph 的条件边（conditional edge）把上面接成图
- [ ] 兜底回答：「资料中未找到相关信息」
- ✅ 里程碑：故意问一个文档里没有的问题，它会重查一轮后老实说不知道，而不是编

### 阶段 3：差异化（MVP 之后，挑 1-2 个，每个 0.5-1 天）
- [ ] **对照实验**：同一批问题，基线 RAG vs DocAgent 各答一遍，结果对比表放进 README（最有面试杀伤力，量化你的价值）
- [ ] **来源引用**：答案末尾标注「依据：文档 X 第 N 段」
- [ ] **多跳问题**：支持需要查两次才能答的问题，展示 Agent 自主多轮检索
- [ ] **简易前端**：FastAPI + Vue3 套壳（复用你已有技术栈）

## 5. 验收标准（做到这些这个项目就「够看」了）

1. 文档里有答案的问题 → 能正确回答并标出来源。
2. 文档里**没有**答案的问题 → 经过重查后明确回答「未找到」，**绝不编造**。
3. 用户问法和文档措辞不一致时 → query 改写后能召回正确片段（举得出例子）。
4. README 里有：动机、架构图、对照 demo 例子、设计取舍说明。
5. 重试有上限，不会死循环；所有外部调用有基本异常处理。

## 7. 面试讲法提示（做完后准备）

被问到时，按这个层次讲：
1. **痛点**：普通 RAG 检索差也硬答会幻觉。
2. **思路**：把 Agent 决策环套在 RAG 外，让它先评估再决定答/重查/认怂。
3. **实现**：LangGraph 状态机，retrieve→grade→decide 条件分叉。
4. **取舍**：举 1-2 个具体决策（如评估为什么用 LLM 不用阈值），讲清代价。
5. **效果**：对照实验，幻觉减少 / 「不知道」识别率提升（拿数字说话，呼应你论文里用指标量化的习惯）。

> 提示：与简历上 VoiceCalendar 的 ReAct 工具调用循环是同一套「决策-行动-观察」思想，面试可串起来讲，体现你对 Agent 范式的整体理解。

## 8. 给 AI 编码助手的协作建议

- 一次只让它实现**一个阶段/一个节点**，跑通再进下一个，别让它一口气生成整个项目（难调试、你也讲不清）。
- 每个节点函数先让它写出**输入/输出的数据结构**，你确认后再写实现。
- 核心的 grade 和 decide 逻辑，**自己读懂每一行**——面试一定会问这块，这是你不能外包给 AI 的部分。
- 让它顺手补单元测试（至少覆盖「有答案/无答案/需改写」三种 case）。
